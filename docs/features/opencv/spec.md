> - 필요하면 두 문서를 같은 커밋 또는 같은 작업 단위에서 함께 수정하라.

## 1. 이 문서는 무엇인가

이 문서는 `OpenCV 기반 프레임 추출 기능`을 비개발자도 이해할 수 있게 설명하는 문서다.

쉽게 말하면 이 기능은:

- 운동 영상을 받아서
- 그 안에서 필요한 장면만 골라
- 분석용 이미지 조각으로 꺼내는 장치다.

가전제품으로 비유하면, 이 기능은 "영상 재료를 넣으면 분석에 쓸 수 있는 장면 조각을 뽑아 주는 기계"라고 보면 된다.

아래 설명은 두 층으로 적는다.

- 쉬운 설명: 비개발자도 이해할 수 있는 비유 중심 설명
- 아키텍처 연동 설명: `architecture.md`나 설계 문서를 수정할 때 바로 연결해서 볼 수 있는 설명

---

## 2. 이 기능을 가전제품으로 보면

### 쉬운 설명

이 기능은 "비디오 프레임 추출기"다.

- 입력구에 비디오 파일을 넣는다.
- 기계 안에서 영상을 한 장면씩 읽는다.
- 사용자가 정한 규칙대로 필요한 장면만 고른다.
- 고른 장면을 밖으로 내보낸다.

이때 밖으로 나가는 결과는 두 종류다.

- 바로 분석기로 넘길 수 있는 메모리 데이터
- 폴더에 저장되는 이미지 파일

즉, 이 기계는 "영상을 분석 가능한 단위로 잘라 주는 중간 장치"다.

### 아키텍처 연동 설명

시스템 구조상 이 기능은 아래 위치에 해당한다.

- 입력 쪽: `controller/jobs.py` 또는 이후 프론트 연동 요청
- 중간 처리: `service/video_reader.py`
- 외부 라이브러리 연결부: `adapter/opencv_adapter.py`
- 출력 연결 대상:
  - `service/pose_inference.py`
  - `service/analysis_pipeline.py`
  - 또는 이미지 저장 경로

즉, 이 기능은 전체 백엔드에서 "비디오를 실제 분석 가능한 단위로 바꾸는 첫 번째 처리 장치"다.

---

## 3. 입력 포트와 출력 포트

### 3.1 입력 포트

### 쉬운 설명

이 기계의 입력 포트는 아래와 같다.

1. 비디오 파일
2. 몇 장면마다 뽑을지 정하는 규칙
3. 이미지를 저장할지 말지 정하는 스위치
4. 색상을 분석용으로 바꿀지 정하는 스위치

현재 프로젝트 기준으로는, 프론트와 백엔드가 완전히 연결되기 전까지 실제 업로드 파일 대신 아래 목업 비디오를 기본 입력으로 사용한다.

- `poseLandmarker_Python/src/video/backSquat.mp4`

즉, 지금은 사용자가 영상을 안 넣어도 내부적으로 샘플 영상을 넣은 상태로 동작한다고 생각하면 된다.

### 아키텍처 연동 설명

현재 입력 관련 기준은 아래와 같다.

- API 입력 진입점: `poseLandmarker_Python/controller/jobs.py`
- 임시 기본 입력 경로: `poseLandmarker_Python/config/config.py`의 `MOCK_VIDEO_PATH`
- 현재 목업 파일:
  - `poseLandmarker_Python/src/video/backSquat.mp4`

현재 `POST /jobs`는 파일 없이도 job을 생성할 수 있고, 내부적으로는 목업 영상 경로를 결과 메타데이터에 넣는다.

---

### 3.2 출력 포트

### 쉬운 설명

이 기계의 출력 포트는 아래와 같다.

1. 프레임 이미지 자체
2. 각 프레임의 번호
3. 각 프레임이 영상의 몇 초 지점인지에 대한 시간 정보
4. 어떤 방식으로 읽었는지에 대한 정보
5. 저장했다면 저장 위치

즉, "사진"만 나오는 것이 아니라 "이 사진이 영상의 어디에서 나온 것인지"까지 같이 딸려 나온다.

### 아키텍처 연동 설명

출력은 개념적으로 아래 두 갈래다.

- 메모리 출력:
  - `ExtractedFrame.image`
  - 후속 `pose_inference` 입력
- 저장 출력:
  - `saved_path`
  - 디스크 이미지 파일

출력 메타데이터는 향후 `schema`에 분리될 수 있지만, 현재 설계 기준으로는 `ExtractedFrame`, `FrameExtractionResult` 구조로 설명하는 것이 가장 직관적이다.

## 4. 이 기능이 실제로 하는 일

### 쉬운 설명

이 기능은 영상을 처음부터 끝까지 그냥 전부 사진으로 바꾸는 기계가 아니다.

필요하면 전부 뽑을 수 있지만, 보통은 아래처럼 규칙을 정해서 뽑는다.

- 모든 프레임 뽑기
- 5프레임마다 1장만 뽑기
- 초당 30장 영상에서 초당 10장만 뽑기
- 3초부터 8초까지만 뽑기

즉, "얼마나 자주", "어느 구간만", "어떤 형태로" 꺼낼지 조절하는 기능이 핵심이다.

### 아키텍처 연동 설명

이 규칙은 `FrameExtractionOptions` 기준으로 정리된다.

- `sampling_mode`
- `every_n_frames`
- `target_fps`
- `start_ms`
- `end_ms`
- `save_images`
- `convert_bgr_to_rgb`

이 옵션들은 주로 `service/video_reader.py`가 해석하고, `adapter/opencv_adapter.py`는 실제 OpenCV 호출만 담당하는 형태가 바람직하다.

---

## 5. 입력된 데이터가 지나가는 길

### 쉬운 설명

이 기능 안에서 데이터는 아래 길을 따라 이동한다.

1. 영상 파일이 들어온다.
2. 영상 열기 장치가 파일을 연다.
3. 영상 정보를 먼저 읽는다.
4. 한 장면씩 꺼낸다.
5. 꺼낸 장면 중 필요한 것만 고른다.
6. 필요하면 색상을 바꾼다.
7. 필요하면 이미지 파일로 저장한다.
8. 결과를 분석기로 넘기거나 결과 묶음으로 반환한다.

이 흐름은 택배 물류창고처럼 생각하면 쉽다.

- 입고: 영상 파일 입력
- 검수: 파일 열기와 메타데이터 확인
- 분류: 샘플링 규칙 적용
- 가공: 색상 변환 또는 리사이즈
- 출고: 저장 또는 분석 파이프라인 전달

### 아키텍처 연동 설명

권장 내부 경로는 아래와 같다.

1. `controller/jobs.py`
   - 요청을 받는다.
2. `service/job_manager.py`
   - 작업을 생성하고 상태를 관리한다.
3. `service/video_reader.py`
   - 프레임 추출 정책을 적용한다.
4. `adapter/opencv_adapter.py`
   - `cv2.VideoCapture`로 실제 비디오를 읽는다.
5. `service/pose_inference.py`
   - 추출된 프레임을 포즈 추론에 사용한다.
6. `service/analysis_pipeline.py`
   - 추론 결과를 분석에 사용한다.

즉, `controller`는 입구, `service/video_reader.py`는 추출 제어부, `adapter/opencv_adapter.py`는 OpenCV와 직접 연결되는 기계 내부 모터라고 보면 된다.

## 6. 입력 데이터는 어떤 형식으로 들어오고, 중간에 어떻게 바뀌는가

### 쉬운 설명

처음에는 "영상 파일" 상태다.

그다음에는 "영상에서 한 장면씩 꺼낸 이미지" 상태가 된다.

그다음에는 "이 장면이 몇 번째이고 몇 초인지가 붙은 분석용 데이터" 상태가 된다.

즉, 데이터는 아래처럼 변한다.

- 비디오 파일
- 프레임 이미지
- 프레임 + 시간 정보 + 저장 정보
- 포즈 분석 입력 데이터

### 아키텍처 연동 설명

형식 변화는 아래처럼 보는 것이 좋다.

1. 파일 경로 또는 업로드 정보
   - 예: `video_path`
2. OpenCV 비디오 핸들
   - 예: `cv2.VideoCapture`
3. 단일 프레임 배열
   - 예: `np.ndarray`
4. 프레임 메타데이터 포함 객체
   - 예: `ExtractedFrame`
5. 배치 또는 iterator 결과
   - 예: `FrameExtractionResult` 또는 `iter_frames()`
6. MediaPipe 입력용 프레임
   - 필요 시 RGB 변환 후 전달

즉, 이 기능의 핵심 형식 변화는 "파일 -> OpenCV 객체 -> 이미지 배열 -> 메타데이터 포함 프레임 객체"다.

## 7. 포트별 상세 설명

### 7.1 비디오 입력 포트

### 쉬운 설명

여기에 들어오는 것은 영상 파일이다.

현재는 이 포트에 직접 넣지 않아도 내부 기본 샘플이 연결돼 있다.

나중에는 아래 둘 다 가능해진다.

- 사용자가 업로드한 영상
- 시스템이 미리 정해 둔 샘플 영상

### 아키텍처 연동 설명

현재 상태:

- API는 업로드를 받을 수 있는 모양을 갖고 있다.
- 하지만 실제 처리 기준 입력은 `MOCK_VIDEO_PATH`다.

향후 연결 완료 시 바뀔 부분:

- `controller/jobs.py`에서 업로드 파일 저장 또는 직접 처리 경로 확보
- `service/video_reader.py`에 실제 `video_path` 전달

---

### 7.2 샘플링 설정 포트

### 쉬운 설명

여기는 "얼마나 자주 장면을 꺼낼지" 정하는 조절 다이얼이다.

예를 들어:

- 모든 장면
- 10장면마다 1개
- 초당 5장
- 2초부터 6초까지만

이 다이얼을 잘 조절하면 속도를 올리고, 불필요한 처리량을 줄일 수 있다.

### 아키텍처 연동 설명

이 포트는 `FrameExtractionOptions`의 샘플링 관련 필드에 대응한다.

- `sampling_mode`
- `every_n_frames`
- `target_fps`
- `start_ms`
- `end_ms`

정책 계산은 `service/video_reader.py`, 실제 읽기는 `adapter/opencv_adapter.py`가 맡는 분리가 적절하다.

---

### 7.3 저장 포트

### 쉬운 설명

이 포트는 "꺼낸 장면을 파일로 저장할지"를 정하는 포트다.

- 꺼내기만 하고 바로 분석기로 넘길 수도 있다.
- 이미지 파일로 폴더에 저장할 수도 있다.

파일 저장은 눈으로 확인하기에는 좋지만, 저장 시간이 추가된다.

### 아키텍처 연동 설명

관련 옵션은 아래와 같다.

- `save_images`
- `output_dir`
- `image_extension`
- `jpeg_quality`

성능상 저장 로직은 코어 추출 로직과 분리하는 것이 좋다.

---

### 7.4 색상 변환 포트

### 쉬운 설명

OpenCV가 읽은 기본 색상 방식과 분석기가 원하는 색상 방식이 다를 수 있다.

그래서 필요하면 색상을 변환한다.

하지만 항상 바꾸는 것은 아니다.

- 저장만 할 때는 굳이 안 바꿔도 된다.
- MediaPipe 분석에 넣을 때만 바꾸는 것이 효율적이다.

### 아키텍처 연동 설명

이 포트는 `convert_bgr_to_rgb`에 대응한다.

권장 정책:

- 저장 목적이면 변환하지 않음
- MediaPipe 투입 직전에만 RGB 변환

이것은 `service/video_reader.py`와 `service/pose_inference.py`의 경계 정책으로 함께 관리하는 것이 좋다.

## 8. 이 기능이 내보내는 결과물의 모양

### 쉬운 설명

결과물은 "사진 여러 장"이 아니라 "정리된 꾸러미"로 보는 것이 좋다.

꾸러미 안에는 보통 아래가 들어 있다.

- 원본 영상 이름 또는 경로
- 원본 영상 FPS
- 전체 프레임 수
- 실제 뽑은 프레임 수
- 각 프레임의 정보

각 프레임 정보 안에는 보통 아래가 있다.

- 몇 번째 프레임인지
- 몇 초 지점인지
- 이미지 자체
- 저장 경로

### 아키텍처 연동 설명

설계 기준 데이터 모델은 아래가 중심이다.

- `FrameExtractionOptions`
- `ExtractedFrame`
- `FrameExtractionResult`

이 구조는 향후 실제 `schema` 파일로 분리되더라도 그대로 기준 문서 역할을 할 수 있다.

## 9. 이 기능이 다른 기능과 연결되는 방식

### 쉬운 설명

이 기계는 혼자 일하는 완성품이 아니라, 다음 기계에 재료를 넘기는 중간 장치다.

연결 대상은 주로 아래 두 개다.

1. 포즈 추론기
2. 운동 분석기

즉, 프레임 추출기는 "영상 분석 공정의 첫 관문"이다.

### 아키텍처 연동 설명

연결 순서는 보통 아래처럼 본다.

- `video_reader`
  -> `pose_inference`
  -> `skeleton_mapper`
  -> `analysis_pipeline`
  -> `llm_feedback`

따라서 이 기능의 출력 형식은 후속 단계가 받기 편한 구조여야 한다.

특히 아래 두 가지를 분리해서 생각해야 한다.

- 프레임 이미지 자체
- 프레임이 어디서 나온 것인지 알려 주는 메타데이터

---

## 10. 현재 임시 동작과 나중 동작의 차이

### 쉬운 설명

현재는 실사용 연결 전이라 "데모 모드"에 가깝다.

지금:

- 내부 샘플 영상 `backSquat.mp4`를 기본 재료로 쓴다.
- 업로드가 없어도 흐름 확인이 가능하다.

나중:

- 사용자가 올린 영상을 진짜 입력으로 받는다.
- 프론트에서 선택한 옵션이 그대로 전달된다.
- 추출된 프레임이 실제 포즈 추론과 분석으로 이어진다.

즉, 지금은 "샘플 재료가 꽂혀 있는 시험 장비", 나중에는 "사용자 재료를 받는 실제 제품"이다.

### 아키텍처 연동 설명

현재 반영된 임시 정책:

- `controller/jobs.py`에서 파일 없이도 job 생성 가능
- `config/config.py`의 `MOCK_VIDEO_PATH` 사용
- 결과 `skeleton.videoInfo.videoSrc`에 목업 비디오 경로 반영

향후 실제 연결 시 변경 포인트:

- 업로드 파일 저장 또는 스트림 처리
- 실제 `video_path` 전달
- `service/video_reader.py`의 실구현 연결
- `adapter/opencv_adapter.py`의 실제 OpenCV 처리 연결

## 11. 성능 관점에서 이 기능이 중요한 이유

### 쉬운 설명

이 기능은 전체 분석 속도를 많이 좌우한다.

왜냐하면:

- 장면을 너무 많이 뽑으면 느려진다.
- 이미지 저장을 많이 하면 느려진다.
- 색상 변환을 자주 하면 느려진다.

즉, 프레임 추출기는 단순한 입구가 아니라 전체 속도를 조절하는 밸브 역할도 한다.

### 아키텍처 연동 설명

성능에 직접 연결되는 주요 선택지는 아래와 같다.

- `sampling_mode`
- `target_fps`
- `save_images`
- `convert_bgr_to_rgb`
- 추후 후보:
  - `preferred_backend`
  - `resize_width`
  - `prefetch_buffer`
  - `decode_mode`
  - `benchmark_profile`

이 내용은 `OpenCV 기반 프레임 추출 기능 아키텍처.md`와 같이 봐야 한다.

## 12. 이 기능이 실패하는 경우

### 쉬운 설명

이 기계는 아래 경우에 정상 작동하지 않을 수 있다.

- 영상 파일이 없다.
- 영상 파일이 손상됐다.
- 영상은 열렸지만 프레임을 읽지 못한다.
- 이미지를 저장하려는데 저장 위치가 잘못됐다.

즉, 재료 자체가 문제이거나, 읽는 과정이 문제이거나, 내보내는 과정이 문제일 수 있다.

### 아키텍처 연동 설명

대표 예외 후보:

- `VideoSourceNotFoundError`
- `VideoOpenError`
- `VideoMetadataError`
- `FrameReadError`
- `FrameWriteError`
- `InvalidSamplingOptionError`

이 예외들은 추후 `service/video_reader.py`와 `adapter/opencv_adapter.py`에 명확히 분리하는 것이 좋다.

## 13. 한 줄 요약

### 쉬운 설명

OpenCV 기반 프레임 추출 기능은 "운동 영상을 넣으면, 분석에 필요한 장면만 골라서 다음 분석 장치로 넘기거나 파일로 저장해 주는 장치"다.

### 아키텍처 연동 설명

시스템 관점에서 이 기능은 `비디오 입력 -> 프레임 추출 -> 포즈 추론/분석 연결`의 첫 번째 실처리 계층이며, `controller -> service/video_reader -> adapter/opencv_adapter -> 후속 service` 흐름의 핵심 시작점이다.
