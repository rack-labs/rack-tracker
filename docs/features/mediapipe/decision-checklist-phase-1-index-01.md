## 1. MVP 범위

결정 질문:

- 1차 구현 범위를 `VIDEO + 2D landmarks only`로 고정할지?
- `landmarks` 외에 어떤 출력을 MVP에 포함할지?

권장안:

- `VIDEO` 모드만 우선 지원
- `landmarks`만 반환
- `world_landmarks`, `segmentation_mask`, 다중 영상 3D 융합은 후속 단계로 보류

추가 확인이 필요한 이유:

- `landmarks`는 현재 웹 오버레이, 기본 스켈레톤 시각화, 기본 분석에 바로 쓰기 좋은 최소 출력이다.
- 다만 MediaPipe는 이 외에도 아래 출력을 다룰 수 있어서, MVP 범위를 어디까지로 자를지 사용자 확인이 필요하다.

선택 가능한 추가 출력:

- `world_landmarks`
  - 카메라 정규화 좌표가 아니라 3D 기준 좌표에 가까운 확장 출력이다.
  - 장점: 향후 3D 분석, 관절 각도 보정, 다중 시점 확장에 유리하다.
  - 단점: 지금 단계에서 후속 소비자가 아직 없으면 직렬화와 스키마 복잡도만 먼저 늘어난다.
- `segmentation_mask`
  - 사람 영역을 픽셀 단위 마스크로 주는 보조 출력이다.
  - 장점: 사람 영역 강조, 배경 제거, 후처리 품질 개선 가능성이 있다.
  - 단점: 현재 파이프라인의 핵심 산출물이 아니고 메모리/직렬화 비용이 커질 수 있다.
- 다중 영상 3D 융합
  - 여러 카메라 결과를 합쳐 더 안정적인 3D 구조를 만드는 후속 아키텍처다.
  - 장점: 장기적으로 더 정교한 분석 가능
  - 단점: 단일 영상 MVP와는 별도 설계가 필요하고 지금 범위를 크게 넓힌다

사용자 확인 질문:

- MVP에서 정말 필요한 출력은 `landmarks`만인지?
- `world_landmarks`를 지금부터 스키마에 실사용 필드로 포함할지, 아니면 문서상 확장 후보로만 둘지?
- `segmentation_mask`는 완전히 제외할지, 옵션만 남길지?

주석: video 모드만 우선 지원하고, 현 단계에선 landmarks만 반환. `world_landmarks`, `segmentation_mask`, 다중 영상 3D 융합은 후속 단계로 보류. `docs/planning/todo.md`에 해당 후속 확장 옵션을 적어둘 것

## 3. 기본 모델 변형

상태: 방향 확정, 문서 보강만 남음, 질문-주석 채널 닫힘

결론 요약:

- 기본 `model_variant`는 `full`로 확정
- 다만 아키텍처 문서와 명세 문서에 "나중에 `lite/full/heavy`로 어떻게 전환하는지"를 함께 명시해야 함
- 즉, 방향은 확정됐고 남은 일은 문서 보강임

결정 질문:

- 기본 `model_variant`를 무엇으로 둘지?

선택지:

- `lite`: 속도 우선
- `full`: 균형형
- `heavy`: 정확도 우선

권장안:

- 기본값은 `full`

추가 설명:

- `lite`
  - 장점: 가장 가볍고 빠르다.
  - 단점: 품질 여유가 작아서 자세 오검출이나 흔들림에 더 민감할 수 있다.
- `full`
  - 장점: 속도와 품질의 균형이 가장 무난하다.
  - 단점: 아주 느린 환경에서는 부담이 될 수 있다.
- `heavy`
  - 장점: 정확도 기대치가 가장 높다.
  - 단점: CPU fallback 시 처리량 저하가 커서 기본값으로 두기 부담스럽다.

아키텍처/명세에 같이 적어둘 보강 포인트:

- 모델 전환은 `PoseInferenceOptions.model_variant` 값을 바꾸는 방식으로 설계한다.
- adapter 계층은 `model_variant`에 따라 어떤 `.task` 번들을 열지 결정한다.
- 즉, 나중에 모델을 바꾸려면 서비스 로직 전체를 바꾸는 것이 아니라:
  - 옵션 기본값
  - 모델 파일 매핑 규칙
  - 배포 시 포함할 `.task` 파일
  이 세 군데를 확인하면 된다.

문서에 적어둘 수정 포인트:

- 아키텍처 문서:
  - `PoseInferenceOptions.model_variant`
  - 초기화 단계의 "model_variant에 맞는 .task 번들 선택"
  - 운영 시 모델 전환 방법
- 명세 문서:
  - 설정 입력값에 `model_variant`
  - 허용값 `lite/full/heavy`
  - 기본값 `full`
  - 모델 변경 시 필요한 파일 준비 조건

주석: 그렇게 진행하되, 코드 어디서 뭘 수정해야 다른 모델로 전환할 수 있는지 아키텍쳐와 명세에 설명 적어놔.

## 4. 초기 실패 처리 정책

상태: 사용자 재확인 필요, 질문-주석 채널 열림

결정 질문:

- 초기 버전에서 어떤 종류의 프레임 문제까지 허용할지?

권장안:

- `입력 프레임 처리 실패`는 작업 실패로 본다.
- `포즈 미검출`은 작업 실패로 보지 않고 정상 결과의 한 종류로 본다.
- 즉, 초기 버전은 "무조건 한 프레임이라도 비면 실패"가 아니라, 실패 유형을 나눠서 본다.

설명:

- 여기서 먼저 구분해야 하는 것은 `프레임 누락`과 `포즈 미검출`이다.
- 이 둘은 운영상 의미가 다르다.

구분 1. 입력 프레임 처리 실패:

- 예:
  - 특정 프레임 이미지가 비어 있음
  - 특정 프레임 디코딩 결과가 손상됨
  - 특정 프레임만 색상 변환 실패
  - 특정 프레임에서 `mp.Image` 생성 실패
  - 특정 프레임 호출에서 예외가 발생해 결과 구조를 만들 수 없음
- 의미:
  - 파이프라인 자체가 해당 프레임을 정상 처리하지 못한 것이다.
  - 이 경우 프레임 타임라인 보존도 흔들릴 수 있다.
- 초기 권장:
  - 작업 실패로 처리

구분 2. 포즈 미검출:

- 예:
  - 사람이 화면 밖으로 벗어남
  - 모션 블러가 심함
  - 가림이 심해서 랜드마크를 못 잡음
  - 프레임은 정상인데 모델 결과만 비어 있음
- 의미:
  - 입력 프레임은 정상이고, 모델이 해당 프레임에서 포즈를 찾지 못한 것이다.
  - 이는 시스템 장애라기보다 입력 품질 또는 모델 한계에 가깝다.
- 초기 권장:
  - 실패로 보지 않고 `pose_detected=False` 같은 정상 결과로 기록

연속성 관점:

- 단순 오버레이나 시각화는 몇 프레임 미검출이 있어도 큰 문제가 아닐 수 있다.
- 하지만 각도 시계열, 속도, 반복 횟수, rep segment 같은 분석은 연속성이 중요하다.
- 다만 여기서 중요한 것은 "모든 프레임에서 포즈가 있어야 한다"보다 `타임라인이 보존되어야 한다`는 점이다.
- 그래서 프레임을 통째로 삭제하는 것보다:
  - 프레임은 유지
  - 미검출은 빈 결과로 기록
  하는 쪽이 후속 분석에 더 안전하다.

운영 해석:

- "허용 가능한 누락 프레임 개수"를 먼저 고정하는 것보다, 실패 유형을 나눠 정책을 정하는 편이 보통 더 안전하다.
- 즉:
  - 입력 프레임 처리 실패는 허용하지 않음
  - 포즈 미검출은 허용
- 이후 실측 데이터가 쌓이면:
  - 미검출 프레임 비율
  - 연속 미검출 길이
  - 분석 품질 저하 기준
  을 보고 경고 임계치를 둘 수 있다.

사용자 확인 질문:

- 1차 버전에서 아래처럼 구분해도 되는지?
- `입력 프레임 처리 실패`는 전체 작업 실패
- `포즈 미검출`은 정상 케이스로 허용하고, 프레임은 유지한 채 빈 결과로 기록
- 추가로, 미검출 프레임 비율이 너무 높을 때 경고만 남길지 여부를 나중 단계 TODO로 둘지?

주석: 
1. 입력 프레임 처리 실패: 작업 실패로 처리. 
구분 2. 포즈 미검출:실패로 보지 않고 `pose_detected=False` 같은 정상 결과로 기록. 추후 단계에서 이 점 상기할 수 있게 todo.md에 적어놔.


## 5. 초기 구현 인터페이스 범위

상태: 방향 확정, TODO 보강만 남음, 질문-주석 채널 닫힘

결론 요약:

- 1차 구현은 `infer()` 중심으로 진행
- 긴 영상 대응용 `iter_infer()`는 후속 단계로 확장
- 후속 확장 경로는 TODO로 문서에 남겨야 함

결정 질문:

- 초기 구현에서 `infer()` 중심으로 단순하게 갈지
- 아니면 처음부터 `iter_infer()`까지 실사용 경로로 만들지

권장안:

- 1차 구현은 `infer()` 우선
- 긴 영상 대응용 `iter_infer()`는 후속 단계에서 확장

추가 설명:

- `infer()`
  - 모든 프레임 결과를 한 번에 모아 반환하는 단순 인터페이스다.
  - 장점: 구현과 테스트가 쉽다.
  - 단점: 긴 영상에서는 메모리 사용량이 커질 수 있다.
- `iter_infer()`
  - 프레임 결과를 순차적으로 흘려보내는 iterator 기반 인터페이스다.
  - 장점: 긴 영상, 스트리밍성 처리, 중간 저장에 유리하다.
  - 단점: 후속 파이프라인도 iterator 친화적으로 바뀌어야 해서 1차 범위가 커진다.

TODO로 적어둘 확장 경로:

- `PoseInferenceService.iter_infer()` 추가
- `skeleton_mapper`가 iterator 입력을 직접 받을지, 중간 accumulator를 둘지 결정
- 긴 영상에서 프레임 단위 저장 또는 chunk 단위 저장 방식 검토
- 중간 진행률 보고와 메모리 사용량 측정 추가
- partial failure 정책과 iterator 경로의 결합 방식 설계

주석: 긴 영상 대응용은 후속 단계에서 확장하고, todo에 가능한 확장 경로를 적어놔.
